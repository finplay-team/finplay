# Spec: 종목과 시세 (종목 시드 · 주식 KRX 데이터 재생 · 코인 실시간 시세 · 장애 처리)

> PRD 근거: MKT-001, MKT-002, MKT-003, MKT-004, MKT-005, MKT-006.

## 개요

주식 16종·코인 12종의 종목 정보를 제공한다. 주식은 KRX 실제 과거 데이터를 검증·정규화해 MySQL에 저장하고 실제 서비스 시간 09:00~15:30 KST에 1배속으로 재생한다. 코인은 업비트 WebSocket으로 실시간 시세를 받아 Redis에 보관한다. 이후 주문(004·005)이 소비할 "유효한 최신 가격"의 유일한 공급원이다.

주식과 코인은 데이터 정본이 다르다 — 주식은 MySQL의 `stock_candles`가 정본이고, 코인은 Redis의 최신 틱이 정본이다. 회원에게는 시장별로 분리된 SSE(Server-Sent Events) 스트림(`/stocks/stream`, `/cryptos/stream`)으로 전달한다.

## 사용자 시나리오

- 사용자는 시장별 종목 목록과 개별 종목의 현재가를 조회한다.
- 사용자는 주식 종목의 1분봉 차트 데이터를 조회한다.
- 사용자는 `/stocks/stream`·`/cryptos/stream` SSE로 시세 변동을 실시간으로 받으며, 주식 화면에는 실제 원본 거래일이 함께 표시된다.
- 사용자는 09:00 개장 직후에는 첫 분봉의 시가로, 09:01부터는 마감된 분봉의 종가로 현재가가 갱신되는 것을 본다.
- 운영자는 KRX 데이터 수집 배치를 실행해 검증을 통과한 1분봉만 저장되고, 재생할 원본 거래일이 장중에 바뀌지 않는 것을 확인한다.
- 운영자는 특정 종목만 데이터가 손상된 날에도 나머지 종목은 정상 개장되는 것을 확인한다.
- 운영자는 업비트 연결이 끊기면 해당 코인 주문이 자동 차단되고, 복구 후 새 틱이 오면 자동 재개되는 것을 확인한다.

## 요구사항

### MKT-001 종목 범위
- [ ] 현재 프론트의 주식 16종·코인 12종을 시드한다.
- [ ] 종목 심볼은 유일하며 시장·호가단위·최소주문금액·거래가능 여부를 가진다.

### MKT-002 주식 실제 과거 데이터 재생
- [ ] KRX 실제 과거 데이터에서 허용된 주식 16종만 추출해 형식·누락을 검증한 뒤 MySQL(`stock_candles`)에 정규화된 1분봉으로 저장한다.
- [ ] 저장된 1분봉을 실제 서비스 시간 09:00~15:30 KST에 1배속(1분마다 1개 공개)으로 재생한다.
- [ ] 모든 회원은 같은 과거 거래일의 같은 분봉을 동시에 본다 — 회원별 복사본을 만들지 않는다.
- [ ] 재생할 원본 거래일은 장 시작 전에 고정되고 당일 장중에는 바뀌지 않는다. 서버가 재시작해도 같은 원본 거래일을 계속 재생한다.
- [ ] 09:00~09:00:59(첫 분봉 구간)에는 첫 분봉의 시가를 현재가로 노출하고 이 구간의 주문도 시가로 체결한다.
- [ ] 09:01부터는 마감이 완료된 마지막 분봉의 종가를 현재가·체결가로 쓴다. 아직 마감하지 않은 분봉의 고가·저가·종가는 노출하지 않는다.
- [ ] 준비된(검증 완료) 원본 거래일 데이터가 없으면 그날 주식 시장을 열지 않는다.
- [ ] 주말·공휴일과 재생 시간 밖에는 주식 시장이 닫힌 상태다.
- [ ] 개장·장중·마감은 주입 가능한 Clock으로 테스트에서 재현할 수 있다.
- [ ] 사용자 화면(SSE 페이로드)에 실제 원본 거래일을 표시한다.

### MKT-003 코인 실시간 시세
- [ ] 업비트 WebSocket에서 12종 최신 틱을 수신한다.
- [ ] 최신 가격·수신시각·연결상태를 Redis에 저장한다.
- [ ] 동일 심볼의 과거 틱이 최신 틱을 덮어쓰지 못한다.

### MKT-004 시세 장애
- [ ] 최신 틱 수신시각이 10초를 초과하거나 연결이 끊기면 해당 코인의 주문 가능 상태가 거부로 바뀐다.
- [ ] 재연결 후 새 틱을 받은 경우에만 주문 가능 상태가 자동 복귀한다.
- [ ] 장애 중 마지막 가격으로 체결하거나 임의 가격으로 몰래 전환하지 않는다.

### MKT-005 데이터 수집과 보관
- [ ] KRX 파일을 읽어 검증·정규화 후 `stock_candles`에 저장하는 수집 컴포넌트가 있다 (추후 API 상품 확인 시 입력 방식만 교체, 재생·주문 로직은 불변).
- [ ] 재생 대상 거래일을 관리하는 재생세션(서비스 날짜·원본 거래일·준비상태·`resolved_at`·실패사유)이 있다. 준비상태는 `PREPARING`·`READY`·`FAILED` 세 가지만 존재한다.
- [ ] 장 시작 전 검증 완료된 최신 거래일을 선택해 재생세션을 `READY`로 전환하고, 당일 바꾸지 않는다.
- [ ] 시장의 개장·마감(OPEN·CLOSED)은 재생세션에 저장하지 않는다 — 준비상태가 `READY`인 세션과 Clock을 조합해 조회 시점에 계산한다.
- [ ] 다음 사유는 명백한 전체 파일 오류로 판정해 그날 재생세션을 `FAILED`로 두고 주식 시장을 열지 않는다 — 이전 가격·임의값 대체 없음: 파일 파싱 불가, 거래일 전체 불일치, 필수 컬럼 부재, 지원하지 않는 파일 구조, 전체 데이터 훼손.
- [ ] 다음은 파일 수집 시 명백한 오류로 검사한다: MVP 허용 16종 여부, 필수 필드 존재, 종목코드 형식, 거래일·분봉시각 형식, 동일 종목·거래일·분봉시각 중복 여부, 분봉시각이 허용된 시장 시간 범위인지, 가격·거래량이 음수가 아닌지, `high >= open`·`high >= close`·`high >= low`·`low <= open`·`low <= close`, 파일의 선언된 거래일과 실제 행의 거래일 일치 여부.
- [ ] 특정 종목의 행에만 위 명백한 구조 오류가 있으면 해당 종목의 행은 거래일 단위로 원자적으로 저장하지 않고, 나머지 정상 종목은 저장한다. 이 경우 수집 이력에 `PARTIAL_SUCCESS`와 문제 종목·원인을 기록한다. 해당 종목은 그 거래일의 유효한 `StockCandle`이 없으므로 가격 조회·주문에서 `PRICE_UNAVAILABLE`로 자연히 거부된다 — 이를 위한 날짜별 종목 전용 상태 테이블이나 컬럼은 만들지 않는다.
- [ ] 분봉 수가 적다는 이유만으로 손상으로 판단하지 않는다 (실제 거래정지·거래 없음·KRX 상품의 빈 분 생략 가능성). 정상 분봉 개수 판단 기준은 확정하지 않는다 (Decision Gate).
- [ ] 수집 시도 이력(성공·부분성공·실패, 실패사유, 수집시각, 파일 중복식별값)을 별도로 기록한다 — 저장에 실패해 `stock_candles`에 행이 없는 경우도 이력에 남는다.
- [ ] `UNIQUE(instrument_id, trading_date, candle_time)`으로 중복 저장을 막고, 수집 작업 재실행이 멱등하다.
- [ ] 한 번 받아들인(SUCCESS·PARTIAL_SUCCESS) 거래일 데이터는 불변으로 취급한다. 같은 거래일·같은 file_hash 재수집은 기존 `StockCandle`·재생세션을 바꾸지 않고 `market_data_imports`에 `SKIPPED_DUPLICATE` 이력만 남긴다. 같은 거래일·다른 file_hash는 기존 데이터·세션을 바꾸지 않고 수집을 거부하며 `market_data_imports`에 `FAILED`(사유 `SOURCE_DATE_HASH_CONFLICT`)로 기록한다. 그 거래일에 `FAILED` 이력만 있다면(받아들인 데이터 없음) 새 파일 재시도를 허용한다.
- [ ] 최근 20영업일만 보관하고, 재생 중인 거래일은 삭제 대상에서 제외하는 정리 작업이 있다.
- [ ] 데이터 출처·실제 거래일·수집시각을 저장한다.

### MKT-006 공개 서비스 데이터 정책
- [ ] 회원에게 원본 데이터 다운로드나 원본 조회 외부 API를 제공하지 않는다.
- [ ] KRX 원본 파일을 공개 저장소에 커밋하지 않는다.

## 비즈니스 규칙

- 주식 시세의 정본은 MySQL `stock_candles`, 코인 시세의 정본은 Redis 최신 틱이다 — 두 시장의 정본은 서로 다르다.
- `stock_replay_sessions`에는 준비상태(`preparation_status`: `PREPARING`·`READY`·`FAILED`)만 저장한다. `OPEN`·`CLOSED`는 DB 상태가 아니라 계산된 시장상태다.
  - `preparation_status != READY` → 주식 시장 이용 불가.
  - `READY` + Clock이 09:00 이전 → `CLOSED`.
  - `READY` + Clock이 09:00~15:30 KST → `OPEN`.
  - `READY` + Clock이 15:30 이후 → `CLOSED`.
  - 주말·공휴일 → `CLOSED`.
- `stock_replay_sessions`의 상태별 필드 규칙 (강제는 `StockReplaySessionScheduler`의 애플리케이션 검증 + 단위 테스트로만 한다 — DB CHECK 제약은 두지 않는다):

  | preparation_status | source_trading_date | resolved_at | failure_reason |
  |---|---|---|---|
  | PREPARING | NULL 가능 | NULL | NULL |
  | READY | 필수 | 필수 | NULL |
  | FAILED | NULL 가능 | 필수 | 필수 |

  - 아직 후보 거래일을 고르기 전 `PREPARING`이면 `source_trading_date`는 NULL일 수 있고, 특정 거래일을 검증하는 중이면 `PREPARING`에서도 값을 가질 수 있다.
  - `FAILED`인데 준비할 데이터 자체를 찾지 못했다면 `source_trading_date`는 NULL, 특정 거래일을 준비하다 실패했다면 그 거래일 값을 가질 수 있다 — 둘 다 허용한다.
  - `resolved_at`은 `READY`·`FAILED`처럼 결과가 결정된 상태에서만 채워진다(`prepared_at`이라는 이름은 성공만 의미하는 것으로 오해될 수 있어 쓰지 않는다).
- 첫 분봉(09:00~09:00:59)은 시가로, 09:01 이후는 마감된 마지막 분봉의 종가로 현재가·체결가를 결정한다. KRX 상품의 분봉 timestamp가 구간 시작 기준인지 종료 기준인지는 상품 확인 후 정확히 매핑한다 — 확인 전에는 구현자가 임의로 해석하지 않는다.
- 코인 시세는 수신시각이 10초를 넘으면 "유효하지 않음"으로 취급한다.
- Redis에는 코인의 최신 시세·수신시각·연결상태만 저장한다 — 원장 데이터 저장 금지 (PRD §6).
- Redis 유실 시에도 MySQL 원장은 보존되며, 새 시세 수신 전까지 해당 코인 주문은 차단된다.
- 검증 실패나 데이터 누락으로 그날 주식 시장을 열지 않을 때, 이전 가격이나 임의 가격으로 몰래 대체하지 않는다.
- 종목별 분봉 누락 임계치(정상적인 거래정지와 데이터 손상의 경계), 그 기준을 저장할 방식, KRX 상품의 timestamp 의미, production 파일 포맷 파서, 정확한 데이터 제공시각, API 수집 여부는 KRX 상품 답변 전까지 확정하지 않는다 (Decision Gate). 샘플 데이터 테스트는 샘플 자체의 기대 결과만 검증하며, 테스트용 숫자를 production 기본값으로 쓰지 않는다.
- 주식 데이터 이용은 PRD C-006(비상업적 교육 목적 공개 서비스, 제한적 제3자 표출)을 따른다.

## 범위 제외

- 주문 검증·체결 로직 (004·005 — 여기서는 "유효한 최신 가격" 조회 계약까지만).
- 실제 업비트 연결 검증, 실제 KRX 파일 수집 검증 (별도 외부 스모크 — 자동 테스트는 Fake Feed·샘플 파일).
- 지정가·호가창·체결강도 등 시세 고도화 (2차 이후).
- 코인 과거 틱·코인 캔들 영구 저장 (PRD 1차 계약의 candles는 주식 1분봉 기반 — 코인은 최신 틱만).
- KRX API 클라이언트 (추후 상품 확인 후 별도 작업).
- 원본 데이터 다운로드 기능, 원본 조회 외부 API (PRD C-006 정책상 1차·2차 모두 제외).
- 종목별 분봉 누락 임계치의 구체적 숫자, 그 기준을 저장하는 별도 스키마, `KrxFileImporter` production 파일 포맷 파서, 정확한 수집 배치 실행시각 (모두 Decision Gate — KRX 답변 후 별도 진행).
- 서버 재기동 없이 진행 중인 재생세션을 강제로 재선택하는 운영 도구 (필요해지면 후속 결정).
- 이미 받아들인 거래일 데이터의 자동 교체, 거래일 전체 원자적 삭제·재삽입, 데이터 버전 관리, 관리자 강제 교체 기능, 장중 데이터 변경 (MVP는 불변 데이터 방식만 지원).

## 완료 조건

- [ ] 종목 목록·단건 조회가 시드된 16+12종을 반환하는 테스트 통과.
- [ ] 고정 Clock으로 개장 전·첫 분봉 구간(시가 노출)·장중(마감된 종가 노출)·마감 후 각각의 장 상태와 현재가 응답 테스트 통과.
- [ ] 재생세션이 `PREPARING`→`READY` 또는 `PREPARING`→`FAILED`로 전환되는 테스트, `READY` 상태에서 Clock에 따라 OPEN·CLOSED가 계산되는 테스트, 서버 재시작 후에도 같은 원본 거래일이 유지되는 테스트, DB에 OPEN·CLOSED가 저장되지 않는다는 테스트 통과.
- [ ] 재생세션 상태별 nullable 규칙 테스트 통과: `PREPARING`+`resolved_at` 존재·`PREPARING`+`failure_reason` 존재·`READY`+`source_trading_date` 없음·`READY`+`resolved_at` 없음·`READY`+`failure_reason` 존재는 모두 거부되고, `FAILED`+`resolved_at` 없음·`FAILED`+`failure_reason` 없음은 거부되며, `FAILED`+`source_trading_date` NULL과 `FAILED`+`source_trading_date` 존재는 둘 다 허용되는 테스트 통과.
