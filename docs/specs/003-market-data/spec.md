# Spec: 종목과 시세 (종목 시드 · 주식 Fixture 재생 · 코인 실시간 시세 · 장애 처리)

> PRD 근거: MKT-001, MKT-002, MKT-003, MKT-004.

## 개요

주식 16종·코인 12종의 종목 정보를 제공하고, 주식은 직전 영업일 1분봉 Fixture를 장중 시간에 1:1 재생하며, 코인은 업비트 WebSocket으로 실시간 시세를 받아 Redis에 보관한다. 이후 주문(004·005)이 소비할 "유효한 최신 가격"의 유일한 공급원이다.

## 사용자 시나리오

- 사용자는 시장별 종목 목록과 개별 종목의 현재가를 조회한다.
- 사용자는 주식 종목의 1분봉 차트 데이터를 조회한다.
- 사용자는 WebSocket으로 시세 변동을 실시간으로 받는다.
- 운영자는 업비트 연결이 끊기면 해당 코인 주문이 자동 차단되고, 복구 후 새 틱이 오면 자동 재개되는 것을 확인한다.

## 요구사항

### MKT-001 종목 범위
- [ ] 현재 프론트의 주식 16종·코인 12종을 시드한다.
- [ ] 종목 심볼은 유일하며 시장·호가단위·최소주문금액·거래가능 여부를 가진다.

### MKT-002 주식 Fixture 재생
- [ ] 저장소에 포함된 직전 영업일 1분봉 Fixture를 09:00~15:30 KST에 1:1로 재생한다.
- [ ] 주말·공휴일과 재생 시간 밖에는 주식 시장이 닫힌 상태다.
- [ ] 개장·장중·마감은 주입 가능한 Clock으로 테스트에서 재현할 수 있다.
- [ ] Fixture는 자동 테스트·시연에 필요한 16종 데이터를 포함한다.

### MKT-003 코인 실시간 시세
- [ ] 업비트 WebSocket에서 12종 최신 틱을 수신한다.
- [ ] 최신 가격·수신시각·연결상태를 Redis에 저장한다.
- [ ] 동일 심볼의 과거 틱이 최신 틱을 덮어쓰지 못한다.

### MKT-004 시세 장애
- [ ] 최신 틱이 기준시간을 초과하거나 연결이 끊기면 해당 코인의 주문 가능 상태가 거부로 바뀐다.
- [ ] 재연결 후 새 틱을 받은 경우에만 주문 가능 상태가 자동 복귀한다.
- [ ] 장애 중 마지막 가격으로 체결하거나 Fixture로 몰래 전환하지 않는다.

## 비즈니스 규칙

- 시세의 정본은 Redis의 최신 틱 하나다 — 주문 체결가는 반드시 이 값에서 나온다.
- 주식 장 상태는 Clock 기준 09:00~15:30 KST · 영업일에만 OPEN이다.
- 코인 시세는 수신시각이 stale 기준을 넘으면 "유효하지 않음"으로 취급한다.
- Redis에는 최신 시세·수신시각·연결상태만 저장한다 — 원장 데이터 저장 금지 (PRD §6).
- Redis 유실 시에도 MySQL 원장은 보존되며, 새 시세 수신 전까지 해당 주문은 차단된다.

## 범위 제외

- 주문 검증·체결 로직 (004·005 — 여기서는 "유효한 최신 가격" 조회 계약까지만).
- 실제 업비트 연결 검증 (별도 외부 스모크 — 자동 테스트는 Fake Feed).
- 지정가·호가창·체결강도 등 시세 고도화 (2차 이후).
- 코인 캔들 데이터 (PRD 1차 계약의 candles는 주식 Fixture 기반 — 코인은 최신 틱만).

## 완료 조건

- [ ] 종목 목록·단건 조회가 시드된 16+12종을 반환하는 테스트 통과.
- [ ] 고정 Clock으로 개장 전·장중·마감 후 각각의 장 상태와 현재가 응답 테스트 통과.
- [ ] Fake Feed로 정상 수신·과거 틱 무시·끊김(주문 차단)·재연결(새 틱 후 재개) 테스트 통과.
- [ ] Redis에 최신 가격·수신시각·연결상태 외 데이터가 저장되지 않음을 확인.
- [ ] `./gradlew build` 통과. 실제 업비트 연결은 미실행 스모크로 구분 보고.
