# Spec: 통합 검증 (핵심 E2E · 요구사항 교차 확인 · Docker 실행)

> PRD 근거: §8 태스크 9, §9 테스트 계획과 완료 기준. 선행: 001~008 전부.
> plan.md·tasks.md는 착수 직전 작성한다.

## 개요

1차 MVP의 완료를 선언하기 위한 검증 태스크다. 기능 추가 없이 핵심 E2E 통합 테스트, PRD 요구사항 ID 교차 확인, Docker 로컬 환경 재현성 검증, 미실행 외부 스모크 구분 보고를 수행한다.

## 사용자 시나리오

- 운영자는 Docker 로컬 환경에서 가입부터 커뮤니티까지 전체 흐름을 재현한다.
- 팀원은 요구사항 ID별로 대응 코드·테스트 위치를 확인한다.

## 요구사항

### 핵심 E2E (PRD §9 — 자동 통합 테스트로 구현)
- [ ] 이메일 또는 Fake OAuth로 가입하면 주식·코인 계좌가 각 1,000만원으로 생성된다.
- [ ] 주식 2회 매수 후 일부 매도 시 FIFO 원가·수수료·현금·보유수량·실현손익이 정확하다.
- [ ] 매수 투자일기를 작성하고 첫 매도 이후 수정이 거부된다.
- [ ] 코인 최신 시세에서 매수·매도가 되고, 시세 연결을 끊으면 주문이 차단된다.
- [ ] 게시물 작성·수정과 댓글 작성·삭제가 되고, 다른 계정의 콘텐츠 변경이 거부된다.

### 교차 확인
- [ ] 1차 요구사항 ID(AUTH·ACCT·MKT·ORD·PORT·JOUR·COM) 각각에 대응 코드·테스트가 있음을 표로 정리한다.
- [ ] `docs/api-routes.md`가 실제 controller와 일치한다.
- [ ] 2차·3차 기능이 1차 코드 경로에 섞여 있지 않다 (PRD C-001).

### Docker 실행
- [ ] Docker Compose로 MySQL·Redis가 기동되고 헬스체크가 통과한다.
- [ ] OAuth 실키 없이 Fake 프로필로 앱이 실행된다.

## 비즈니스 규칙

- Mock/Fake 통과를 실제 외부 연동 성공으로 표현하지 않는다 (PRD C-005).
- 실행한 검증·미실행 외부 스모크(실 업비트·카카오·네이버)·남은 위험을 구분해 보고서로 남긴다.

## 범위 제외

- 신규 기능·리팩터링 (검증 중 발견된 결함 수정은 해당 spec으로 돌려 처리).
- 부하테스트·동시성 검증 (2차).
- CI 구성과 배포 후 스모크 (`010-deployment`로 이관 — 첫 배포가 이 태스크보다 앞서기 때문).

## 완료 조건

- [ ] PRD §9 핵심 E2E 9단계가 자동 통합 테스트로 통과한다.
- [ ] 요구사항 ID 교차표가 작성되고 공백이 없다.
- [ ] `./gradlew build` + Docker 환경 E2E 재현이 모두 통과한다.
- [ ] 검증 보고서(실행/미실행/위험 구분)가 남는다.
