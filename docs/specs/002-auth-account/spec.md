# Spec: 인증과 계좌 (회원가입 · JWT · OAuth · 시장별 계좌 생성)

> PRD 근거: AUTH-001, AUTH-002, AUTH-003, AUTH-004, ACCT-001.

## 개요

이메일 인증을 완료한 방문자만 회원가입할 수 있게 하고, 로그인과 카카오·네이버 OAuth를 제공하며, 가입 시 주식·코인 계좌 2개를 시드머니 1,000만원씩으로 생성한다. 이후 모든 인증 필요 API의 기반이 된다.

## 사용자 시나리오

- 방문자는 이메일로 인증번호를 받아 확인하고, 발급된 가입 토큰과 함께 닉네임·비밀번호·약관 동의를 보내 가입하며, 즉시 JWT 토큰을 받는다.
- 가입한 순간 주식 계좌와 코인 계좌가 각각 현금 10,000,000원으로 만들어져 있다.
- 사용자는 로그인해 Access Token과 Refresh Token을 받고, 만료 시 재발급받는다.
- 방문자는 카카오 또는 네이버로 로그인해 회원이 되며, 같은 이메일의 기존 회원이면 소셜 계정이 그 회원에 연결된다.
- 사용자는 로그아웃하면 기존 Refresh Token을 더 이상 쓸 수 없다.

## 요구사항

### AUTH-001 이메일 회원가입
- [ ] 입력은 이메일, 중복되지 않는 닉네임, 8자 이상 비밀번호, 약관 동의, `signupVerificationToken`이다.
- [ ] 비밀번호는 단방향 해시로 저장한다.
- [ ] 이메일 또는 닉네임 중복은 409 `DUPLICATE_RESOURCE`로 거부한다.
- [ ] 토큰이 없거나 만료됐거나 이미 사용됐으면 409 `EMAIL_VERIFICATION_REQUIRED`로 거부한다.
- [ ] 토큰에 묶인 이메일과 요청 이메일이 다르면 409 `EMAIL_VERIFICATION_REQUIRED`로 거부한다.
- [ ] 성공 시 토큰 소비와 회원·주식계좌·코인계좌 생성을 같은 트랜잭션에서 한 번만 처리하고 JWT 토큰을 반환한다.

### AUTH-002 로그인·토큰
- [ ] Access Token과 회전형 Refresh Token을 사용한다.
- [ ] Refresh Token 원문은 저장하지 않고 해시만 저장한다.
- [ ] 로그아웃 또는 재발급 시 이전 Refresh Token을 폐기한다 (폐기된 토큰 재사용은 401).
- [ ] 인증 필요 API는 `Authorization: Bearer <accessToken>`을 요구하며, 없거나 만료면 401 `UNAUTHORIZED`.

### AUTH-003 카카오·네이버 OAuth
- [ ] `KAKAO`, `NAVER`를 공통 OAuth 어댑터로 구현한다.
- [ ] 최초 소셜 로그인은 회원과 두 계좌를 같은 트랜잭션에서 생성한다.
- [ ] 제공자 이메일이 기존 이메일 회원과 같으면 그 회원에 소셜 계정을 연결한다 (계좌 추가 생성 없음).
- [ ] 같은 `provider + providerUserId`는 한 회원에게만 연결된다 (중복 시 409).
- [ ] 자동 테스트는 Fake OAuth 제공자를 사용한다. 실제 제공자 검증은 별도 스모크 테스트로 구분한다.

### AUTH-004 이메일 인증 (가입 선행)
- [ ] 인증번호는 6자리 숫자, 유효시간 5분, 입력 시도 최대 5회다.
- [ ] 시도 5회를 초과하면 해당 인증번호를 즉시 무효화하고 429 `TOO_MANY_REQUESTS`로 응답한다.
- [ ] 재발송은 60초 간격, 같은 이메일에 1시간 5회·하루 10회로 제한한다. 초과 시 429 `TOO_MANY_REQUESTS`다.
- [ ] 재발송하면 이전 인증번호를 즉시 무효화한다 — 유효한 인증번호는 항상 최대 1개다.
- [ ] 인증번호 불일치·만료·미발급은 400 `EMAIL_VERIFICATION_FAILED`로 응답한다.
- [ ] 인증 성공 시 30분 유효한 일회용 `signupVerificationToken`을 발급한다.
- [ ] 인증번호는 `EMAIL_VERIFICATION_SECRET` 기반 HMAC-SHA-256, 가입 토큰은 SHA-256으로 저장한다 (원문 미저장).
- [ ] 이미 가입된 이메일의 인증 요청은 409 `DUPLICATE_RESOURCE`로 거부한다.
- [ ] 로컬·테스트는 `FakeEmailSender`, 운영은 `ResendEmailSender`를 쓴다.

### ACCT-001 시장별 계좌
- [ ] 회원당 `STOCK` 계좌 1개, `CRYPTO` 계좌 1개만 존재한다 (`UNIQUE(user_id, market)`).
- [ ] 각 계좌의 최초 현금은 10,000,000원이다.
- [ ] 계좌 간 이체는 지원하지 않는다.

## 비즈니스 규칙

- 이메일 인증 단계에서는 `users`·`accounts` 행을 만들지 않는다 — 두 테이블은 가입 트랜잭션에서만 생성된다.
- 가입 트랜잭션이 닉네임 중복 등으로 실패하면 가입 토큰 소비도 함께 롤백되어, 남은 유효시간 안에 같은 토큰으로 재시도할 수 있다.
- 인증번호와 가입 토큰은 원문을 저장하지 않는다. 해시는 저장소 유출에 대한 심층방어이며 주 방어는 5분 만료와 5회 시도 제한이다.
- 가입(이메일·소셜 공통) 성공 시 계좌 2개(STOCK·CRYPTO)가 반드시 함께 생성된다 — 부분 성공 없음.
- 계좌 초기 현금은 항상 10,000,000원이다.
- 비밀번호·Refresh Token은 원문을 저장하지 않는다 (각각 BCrypt·해시).
- Refresh Token은 1회용이다 — 재발급에 쓰인 토큰은 즉시 폐기된다.
- 소셜 계정 1개(provider + providerUserId)는 회원 1명에게만 속한다.

## 범위 제외

- 계좌 잔고·요약 조회 API (006 조회 spec).
- 회원 탈퇴, 프로필 이미지, 파일 업로드 (PRD 1차 명시적 제외).
- 만료된 이메일 인증 데이터 정리 배치 (MVP 제외 — 데이터량이 적어 후속으로 미룸).
- 실제 카카오·네이버 키 연동 검증, 실제 Resend 발송·수신 (별도 외부 스모크 — PRD §9).

## 완료 조건

- [ ] 인증번호 요청 → 확인 → 가입 토큰으로 가입까지의 통합 테스트가 `FakeEmailSender`로 통과.
- [ ] 인증번호 5회 초과 시 429, 재발송 시 이전 코드 무효화, 만료 코드 400 테스트 통과.
- [ ] 같은 가입 토큰 재사용이 409 `EMAIL_VERIFICATION_REQUIRED`로 거부되는 테스트 통과.
- [ ] 닉네임 중복으로 가입이 실패한 뒤 같은 토큰으로 재시도가 성공하는 테스트 통과.
- [ ] `RESEND_API_KEY`·`EMAIL_FROM` 없이 `./gradlew build`와 로컬 기동이 성공.
- [ ] 가입 성공 시 회원 1 + 계좌 2(STOCK·CRYPTO, 각 1,000만원)가 정확히 한 번 생성되고 토큰이 반환되는 통합 테스트 통과.
- [ ] 이메일·닉네임 중복 각각 409 응답 테스트 통과.
- [ ] 로그인→재발급→이전 Refresh Token 재사용 거부(401) 테스트 통과.
- [ ] Fake OAuth 신규 가입(계좌 2개 생성)·기존 이메일 병합(계좌 추가 없음) 테스트 통과.
- [ ] 인증 없는 보호 API 접근이 401 공통 오류 포맷으로 응답.
- [ ] `./gradlew build` 통과.
