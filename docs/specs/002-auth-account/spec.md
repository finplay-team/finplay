# Spec: 인증과 계좌 (회원가입 · JWT · OAuth · 시장별 계좌 생성)

> PRD 근거: AUTH-001, AUTH-002, AUTH-003, ACCT-001.

## 개요

이메일 회원가입과 로그인, 카카오·네이버 OAuth를 제공하고, 가입 시 주식·코인 계좌 2개를 시드머니 1,000만원씩으로 생성한다. 이후 모든 인증 필요 API의 기반이 된다.

## 사용자 시나리오

- 방문자는 이메일·닉네임·비밀번호·약관 동의로 가입하고, 즉시 JWT 토큰을 받는다.
- 가입한 순간 주식 계좌와 코인 계좌가 각각 현금 10,000,000원으로 만들어져 있다.
- 사용자는 로그인해 Access Token과 Refresh Token을 받고, 만료 시 재발급받는다.
- 방문자는 카카오 또는 네이버로 로그인해 회원이 되며, 같은 이메일의 기존 회원이면 소셜 계정이 그 회원에 연결된다.
- 사용자는 로그아웃하면 기존 Refresh Token을 더 이상 쓸 수 없다.

## 요구사항

### AUTH-001 이메일 회원가입
- [ ] 입력은 이메일, 중복되지 않는 닉네임, 8자 이상 비밀번호, 약관 동의다.
- [ ] 비밀번호는 단방향 해시로 저장한다.
- [ ] 이메일 또는 닉네임 중복은 409 `DUPLICATE_RESOURCE`로 거부한다.
- [ ] 성공 시 회원·주식계좌·코인계좌를 같은 트랜잭션에서 한 번만 생성하고 JWT 토큰을 반환한다.

### AUTH-002 로그인·토큰
- [ ] Access Token과 회전형 Refresh Token을 사용한다.
- [ ] Refresh Token 원문은 저장하지 않고 해시만 저장한다.
- [ ] 로그아웃 또는 재발급 시 이전 Refresh Token을 폐기한다 (폐기된 토큰 재사용은 401).
- [ ] 인증 필요 API는 `Authorization: Bearer <accessToken>`을 요구하며, 없거나 만료면 401 `UNAUTHORIZED`.

### AUTH-003 카카오·네이버 OAuth
- [ ] `KAKAO`, `NAVER`를 공통 OAuth 어댑터로 구현한다.
- [ ] 최초 소셜 로그인은 회원과 두 계좌를 같은 트랜잭션에서 생성한다.
- [ ] 제공자 이메일이 기존 이메일 회원과 같으면 그 회원에 소셜 계정을 연결한다 (계좌 추가 생성 없음).
- [ ] 같은 `provider + providerUserId`는 한 회원에게만 연결된다 (중복 시 409).
- [ ] 자동 테스트는 Fake OAuth 제공자를 사용한다. 실제 제공자 검증은 별도 스모크 테스트로 구분한다.

### ACCT-001 시장별 계좌
- [ ] 회원당 `STOCK` 계좌 1개, `CRYPTO` 계좌 1개만 존재한다 (`UNIQUE(user_id, market)`).
- [ ] 각 계좌의 최초 현금은 10,000,000원이다.
- [ ] 계좌 간 이체는 지원하지 않는다.

## 비즈니스 규칙

- 가입(이메일·소셜 공통) 성공 시 계좌 2개(STOCK·CRYPTO)가 반드시 함께 생성된다 — 부분 성공 없음.
- 계좌 초기 현금은 항상 10,000,000원이다.
- 비밀번호·Refresh Token은 원문을 저장하지 않는다 (각각 BCrypt·해시).
- Refresh Token은 1회용이다 — 재발급에 쓰인 토큰은 즉시 폐기된다.
- 소셜 계정 1개(provider + providerUserId)는 회원 1명에게만 속한다.

## 범위 제외

- 계좌 잔고·요약 조회 API (006 조회 spec).
- 회원 탈퇴, 프로필 이미지, 파일 업로드 (PRD 1차 명시적 제외).
- 이메일 인증(verification) 발송·확인 흐름 — PRD §3 로드맵에 언급되나 §4 요구사항 ID·수용 기준이 없어 제외. 필요하면 PRD 후속 결정 후 별도 spec.
- 실제 카카오·네이버 키 연동 검증 (별도 외부 스모크 — PRD §9).

## 완료 조건

- [ ] 가입 성공 시 회원 1 + 계좌 2(STOCK·CRYPTO, 각 1,000만원)가 정확히 한 번 생성되고 토큰이 반환되는 통합 테스트 통과.
- [ ] 이메일·닉네임 중복 각각 409 응답 테스트 통과.
- [ ] 로그인→재발급→이전 Refresh Token 재사용 거부(401) 테스트 통과.
- [ ] Fake OAuth 신규 가입(계좌 2개 생성)·기존 이메일 병합(계좌 추가 없음) 테스트 통과.
- [ ] 인증 없는 보호 API 접근이 401 공통 오류 포맷으로 응답.
- [ ] `./gradlew build` 통과.
