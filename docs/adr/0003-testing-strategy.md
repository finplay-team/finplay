# ADR-0003: 테스트 전략 — mock 단위 테스트만으로 끝내지 않는다

- 상태: 승인됨
- 날짜: 2026-07-22

## 맥락

mock 기반 단위 테스트만 있으면 각 클래스는 통과해도 통합 시점(실제 쿼리, 직렬화, 트랜잭션 경계)에서 오류가 난다. 반대로 모든 걸 통합 테스트로 하면 느려서 아무도 돌리지 않는다.

## 결정

3단계 테스트 피라미드를 사용한다.

| 레벨 | 도구 | 대상 | 비중 |
|---|---|---|---|
| 단위 | JUnit5 + Mockito | 서비스 비즈니스 로직 (의존성 mock) | 다수 |
| 슬라이스 | `@DataJpaTest`, `@WebMvcTest` | JPA 쿼리 실제 실행, 컨트롤러 직렬화·검증·예외 매핑 | 중간 |
| 통합 | `@SpringBootTest` + Testcontainers (MySQL 8.4) | 핵심 사용자 시나리오 전체 흐름 | 소수 |

- **H2 인메모리 DB는 쓰지 않는다.** MySQL과 SQL 방언이 달라 통과해도 운영에서 깨진다. 통합·슬라이스 테스트 모두 Testcontainers MySQL을 쓴다 (`TestcontainersConfiguration` import).
- 컨테이너 이미지는 `mysql:8.4`로 고정한다 (`latest` 금지).
- 테스트용 MySQL 컨테이너는 공유 싱글턴 하나만 쓴다 (`TestcontainersConfiguration`의 static 인스턴스). 테스트 컨텍스트마다 재기동하지 않는다.
- 새 기능의 완료 조건: 서비스 단위 테스트 + 해당 레벨 슬라이스 테스트. 스펙의 핵심 시나리오면 통합 테스트 1개 추가.

## 결과

- 통합 테스트 실행에 Docker가 필요하다. CI에는 Docker가 기본 제공되므로 문제없고, 로컬에서 Docker가 없으면 `./gradlew test --tests "*Test"` 등으로 단위만 선별 실행할 수 있다.
- 테스트가 다소 느려지는 대신 "붙이면 깨지는" 유형의 결함을 머지 전에 잡는다.
